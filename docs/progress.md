# Live progress behavior and feasibility

## Conclusion

Portable live phase, elapsed-time, activity, and per-copy rsync progress is feasible, but Kubernetes and CSI provide no portable, reliable overall percentage for this replacement-based shrink workflow.

The portable boundary is an observational display: name the current phase and sub-phase, show total and phase elapsed time, report the latest Kubernetes activity, and relay rsync's own output for each copy independently. It must not change the workflow's decisions, waits, cancellation, Lease ownership, recovery checkpoints, cleanup, destructive-operation ordering, or command semantics. The local `--quiet` flag disables only this observational display for the current invocation.

## Implemented behavior and output contract

The workflow publishes the canonical display phases below while retaining the existing two-second lifecycle polling gates. Every snapshot contains the current phase, total elapsed time since the current command invocation, elapsed time since that display phase began, and the latest available activity. Go's monotonic time component is used for elapsed calculations, phase timing resets only when the canonical phase changes, and every `--resume` invocation starts a fresh total timer. No elapsed history is persisted or reconstructed.

`IOStreams.Out` is classified at runtime. On a TTY, progress is bounded to the terminal width and one line is safely cleared and redrawn. Before durable output or stderr diagnostics, that line is terminated so redraws cannot corrupt text. On a pipe or redirected file, snapshots are bounded, complete, newline-delimited records and contain no carriage-return or ANSI redraw controls. Phase changes are queued promptly, unchanged state receives a low-rate heartbeat, and bursty rsync records are coalesced and rate-limited. Rendering is asynchronous and errors are ignored, so polling callbacks do not return display failures into workflow control flow.

The root command's local Boolean `--quiet` flag defaults to false, has no shorthand, and is invocation-only. It suppresses stage, phase, heartbeat, activity, cleanup, and streamed rsync snapshots. It does **not** imply `--yes`; suppress the execution plan; suppress the confirmation prompt, source-usage summary, dry-run result, final success or recovery result, help, or version output; suppress stderr warnings or errors; alter Kubernetes requests; or change exit status. It works for normal, dry-run, and resumed invocations and is not written to the recovery ConfigMap.

Only the two copy Jobs use a best-effort `Follow` stream for Pod logs. Copy output is parsed across arbitrary read boundaries, with both carriage returns and newlines treated as record delimiters and a final partial record retained. Records and rendered activity are bounded and terminal control characters are removed. The displays are independently labeled `copy 1 of 2: source to temporary` and `copy 2 of 2: temporary to source`; recognized percentages are explicitly labeled per-copy and reset for copy-back. Stream setup, reading, formatting, and closure failures fall back to Job and Pod lifecycle activity. Terminal Job conditions remain the copy authority, and the existing bounded post-failure log retrieval and cleanup path remains separate and unchanged. Verification and inspection logs are still consumed only after completion for their existing safety parsing.

The persisted operation phases remain coarse recovery checkpoints, not display progress. In particular, `Prepared` is written before consumer scale-down and spans scale-down, source unmounting, usage inspection, temporary PVC creation, the first copy, and the first verification. Later checkpoints protect the source-delete, source-recreate, copy-back, and restore boundaries. Display phase, activity, elapsed time, rsync records, and `--quiet` are never added to [`internal/operation.State`](../internal/operation/state.go).

## Implementation requirements

The implementation uses the Kubernetes client baseline pinned at `v0.36.2` in [`go.mod`](../go.mod) and only the permissions already documented in [`examples/rbac.yaml`](../examples/rbac.yaml). Those permissions already allow the required PVC, Pod, Job, Deployment scale, ConfigMap, Lease, and `pods/log` reads. The portable design needs no watches, no Events access, no additional RBAC, and no new dependency.

1. Emit stable phase and sub-phase names matching the matrix. Keep them observational: the existing API results and readiness gates remain authoritative. A progress-rendering failure must not fail, delay, retry, reorder, or skip workflow work.
2. Show **total elapsed time** since the current command invocation and **phase elapsed time** since the current displayed phase began, using a monotonic clock. On `--resume`, start a new invocation timer rather than inventing elapsed history that was never persisted. Do not turn elapsed time into an ETA.
3. On the existing two-second polling cycle, publish a heartbeat containing the current phase and **latest activity**: for example, the remaining active consumer Pods, observed PVC existence and PVC phase, a Pod waiting reason, or the latest Job condition. Emit changes immediately when practical and retain a low-rate heartbeat when the observation is unchanged.
4. Use **Pod-log streaming only during the two rsync copy stages**. The existing `get` permission on `pods/log` is sufficient for Pod-log streaming, so this requires no additional RBAC. Keep the two-second Job and Pod polling as the lifecycle authority. Failure or delay of the log stream must degrade to phase/activity output rather than alter the copy result.
5. Label the two rsync displays explicitly, such as `copy 1 of 2: source to temporary` and `copy 2 of 2: temporary to source`. Label every rsync percentage **per-copy**. Reset it for the second invocation and never average, add, weight, or otherwise combine the values into an overall percentage.
6. Treat rsync log data as a chunked text stream. Preserve partial records across chunks, recognize both newline and carriage-return updates, and rate-limit rendering. A TTY may redraw one line; a non-TTY should receive bounded line-oriented snapshots. This display policy is observational and must not change output parsing used for verification.
7. Close any copy-log stream through the same cancellation context as the copy while preserving the existing bounded post-failure log retrieval and cleanup path. Do not change Ctrl-C behavior, Lease cancellation, restoration decisions, or cleanup ordering.

This recommendation adds no snapshots, clones, readiness gates, Events, watches, permissions, manifests, dependencies, or driver-specific requirements. Optional CSI-driver or backend metrics can be displayed in a clearly separate diagnostic field when an operator already has them, but the portable phase model must neither query nor depend on them.

## Accuracy and limitations

Kubernetes lifecycle fields are operational observations rather than data-migration progress. A PVC becoming Bound, a Pod becoming Running, a container reporting a waiting reason, or a Job gaining a Complete condition says what Kubernetes currently observes; none reports how many migration bytes remain. The Kubernetes documentation similarly describes [Pod phase as a high-level lifecycle summary](https://kubernetes.io/docs/concepts/workloads/pods/pod-lifecycle/), [Jobs through lifecycle and terminal conditions](https://kubernetes.io/docs/concepts/workloads/controllers/job/), [PVC provisioning and binding](https://kubernetes.io/docs/concepts/storage/persistent-volumes/), and [Leases as coordination objects](https://kubernetes.io/docs/concepts/architecture/leases/).

There is no authoritative byte count shared by all stages, no defensible stage weighting, and no portable ETA. Provisioning, scheduling, attaching, mounting, deleting, two copies, two checksum scans, controller reconciliation, and cleanup have environment-dependent costs. Counting completed phases would make fast and slow phases look equal; assigning weights would merely encode assumptions about a particular cluster.

The final `du` result is fit-safety data, not a reliable rsync denominator. `du` reports filesystem usage according to its own allocation semantics, while rsync reports work according to the file list and transfer rules of that invocation. Exclusions, sparse files, hard links, metadata, filesystem allocation, and backend compression or deduplication can make those values diverge. The inspection also precedes two distinct copy invocations. It must not be presented as an authoritative byte count or used to manufacture an overall percentage.

`--info=progress2` remains useful but limited. Its availability and exact format depend on the user-selected image and rsync version. Its scope depends on the effective rsync arguments, including exclusions and any options that affect traversal or transfer behavior. Sparse files and metadata work can make displayed bytes differ from physical I/O or elapsed effort. Container stdout, the runtime, and the Kubernetes log path may buffer output, and progress2 commonly uses carriage-return-formatted updates that do not arrive as one complete line per update. These limitations make it a per-copy, tool-reported observation only.

An rsync percentage is not resumable progress. The persisted phases in [`internal/operation/state.go`](../internal/operation/state.go) are coarse recovery checkpoints; they do not store rsync state or percentages. Because `Prepared` covers multiple activities through the first verification, an interrupted copy may restart in a new Job rather than resume from a displayed percentage. A later invocation may skip already-matching files, but that does not turn the prior percentage into a durable checkpoint.

Usage inspection and both checksum verifications can show phase elapsed time, Pod or Job activity, and a heartbeat, but no generic live byte percentage. The final inspection value and verification output or `--stats` are completion-time signals. Checksum completion demonstrates the configured comparison completed; it does not reveal a portable fraction while hashing is in progress.

PVC expansion and CSI `ExpandVolume` operations are inapplicable here and must not be presented as progress. Kubernetes expansion changes a claim toward a larger capacity; this shrink workflow instead provisions replacement PVCs and migrates data. The [Kubernetes volume expansion documentation](https://kubernetes.io/docs/concepts/storage/persistent-volumes/#expanding-persistent-volumes-claims) and the official [Container Storage Interface specification](https://github.com/container-storage-interface/spec/blob/master/spec.md) define expansion and storage RPC outcomes but no aggregate progress contract for this replacement sequence.

Snapshots and clones do not solve the aggregate-progress problem. A snapshot can expose readiness, and a cloned PVC can expose ordinary claim lifecycle, but neither Kubernetes nor CSI standardizes a portable percentage spanning backend copy-on-write work, provisioning, mounts, verification, restoration, and cleanup. Support and implementation are driver-dependent. The [Kubernetes snapshot documentation](https://kubernetes.io/docs/concepts/storage/volume-snapshots/) and [CSI volume cloning documentation](https://kubernetes.io/docs/concepts/storage/volume-pvc-datasource/) are useful for characterizing those limitations only; this recommendation does not add snapshots or clones to the workflow.

Driver-specific dashboards, storage-system task states, throughput counters, or estimated completion values are optional and non-portable. They may explain why a PVC is pending in a known environment, but they must be labeled as backend diagnostics, must not target a particular driver or backend in the portable design, and must never override Kubernetes object identity, Job success, checksum verification, or the workflow's recovery and safety decisions.
